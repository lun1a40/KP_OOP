# Принятые решения (ADR-style)

Короткие записи о ключевых выборах в архитектуре и почему именно так, а не иначе. Это нужно, чтобы на защите можно было обосновать каждое решение.

---

## ADR-1: Веб-интерфейс вместо консоли или Qt

**Решение:** UI реализован как HTML/CSS/JS, который отдаёт встроенный HTTP-сервер.

**Альтернативы:**
- Консоль — скучно, не оценивается высоко
- Qt/WinForms — требует установки runtime, сложнее сдавать
- Web — самостоятельно, один бинарь + папка статики

**Почему:** требование курсовой («интерфейс — сделаем как сайт»). Дополнительный плюс — можно показать REST API отдельно от UI.

---

## ADR-2: cpp-httplib вместо Crow/Drogon/CGI

**Решение:** [cpp-httplib](https://github.com/yhirose/cpp-httplib) — один header-файл.

**Почему:**
- **Нулевые зависимости** — никаких пакетных менеджеров (vcpkg/conan), никаких *.lib, никакого Boost.
- **Простота защиты** — комиссии не нужно объяснять асинхронность.
- На Windows работает «из коробки» (внутри `#pragma comment(lib, "Ws2_32.lib")`).

**Минусы:** синхронный, не выдержит production нагрузку. Для курсача — иррелевантно.

## ADR-3: JSON-файлы вместо SQLite

**Решение:** хранение данных в `data/*.json`, загрузка в память при старте, перезапись при изменениях.

**Почему:**
- **Демонстрация ООП-сериализации** — виртуальный `toJson()` и фабричный метод по полю `category` хорошо показывают полиморфизм.
- Не нужно объяснять SQL, foreign keys, миграции.
- Файлы открываются глазами, легко править вручную при отладке.

**Когда стало бы плохо:** если бы записей были тысячи и требовалась эффективная фильтрация. Не наш случай.

## ADR-4: Деньги в `int64_t` копейках, не `double`

**Решение:** обёртка `util::Money` хранит `int64_t kopecks_`.

**Почему:** `double` неточен для накопительных расчётов амортизации. На длинном жизненном цикле фонда (10-15 лет, месячные начисления) ошибки округления накапливаются и портят отчёт. С целыми числами всё детерминированно.

---

## ADR-5: Виртуальный `calculateDepreciation` на самом классе, а не отдельная Стратегия

**Решение:** каждый наследник `FixedAsset` сам считает амортизацию через `override virtual`.

**Альтернатива:** паттерн «Стратегия» — `DepreciationStrategy` с подклассами `LinearStrategy`, `MileageStrategy`, и т.д., которые внедряются в `FixedAsset`.

**Почему выбрана прямая виртуальная:**
- **Полиморфизм виден сразу.** На защите легче ткнуть пальцем в `override` в четырёх классах.
- Алгоритм амортизации жёстко связан с типом фонда (здание не считают «по пробегу»), так что подменять стратегию некуда.

**Где Стратегия всё-таки появилась:** в `ReportFormatter` (JSON / CSV) — там разные варианты вывода действительно взаимозаменяемы.

## ADR-6: `std::unique_ptr<FixedAsset>` вместо плоского хранения

**Решение:** репозиторий хранит `std::unordered_map<int, std::unique_ptr<FixedAsset>>`.

**Почему:**
- `FixedAsset` абстрактный — не положить «по значению»
- `unique_ptr` исключает утечки, исключает копирование (важно для отладки)
- При сериализации просто итерируемся: `for (auto& [id, ptr] : map) ptr->toJson();`

`shared_ptr` не нужен — единый владелец (репозиторий), сервисам передаётся `const FixedAsset&` или `*`.

## ADR-7: Кастомный `util::Date`, а не `std::chrono::year_month_day`

**Решение:** свой класс `Date`.

**Почему:** `std::chrono` C++20-возможности (`year_month_day`) есть, но в MSVC 2019 поддержка частичная, парсинг/форматирование громоздкое. Свой класс из ~50 строк проще объяснить и контролировать.

---

## ADR-8: Vanilla JS вместо React/Vue

**Решение:** обычный HTML + `fetch` без фреймворков.

**Почему:**
- Курсовая на C++ — фокус не должен размываться на JS-toolchain
- Нет node_modules, нет сборки — открыл .html, всё работает
- Логики на фронте — пара сотен строк, фреймворк не нужен

---

## ADR-9: Сборка через CMake + Docker, а не вручную собранный VS-проект

**Решение:** канонической системой сборки выбран **CMake**, запуск — через
**многоступенчатый Docker-образ** + `docker compose`.

**Почему:**
- **Воспроизводимость** — `docker compose up --build` поднимает сервер одинаково на любой машине, без «у меня не собралось».
- **Один источник правды** — `CMakeLists.txt` глобит `src/`, не нужно вручную добавлять каждый файл в `.vcxproj`.
- **VS не теряется** — CMake генерирует `.sln` (`-G "Visual Studio 17 2022"`), так что Windows-сдача остаётся возможной.
- Код проверяется на g++ (Docker) и clang (macOS) — гарантия портируемости.

**Минус:** ещё один инструмент (CMake) в стеке. Оправдано: он стандартный и заменяет ручную возню с проектом VS.

## ADR-10: Уточнения реализации (закрытие нестыковок проектной документации)

В ходе разработки уточнены несколько мест, расходившихся между документами:

- **`Vehicle::calculateDepreciation`** — добавлен потолок в `initialCost` (общий хелпер `depreciationFromRatio` клампит долю в `[0,1]`). Иначе при пробеге выше расчётного остаточная стоимость уходила в минус.
- **`JsonMapper`** — это **фабрика** `assetFromJson` (полиморфное создание по `category`) **плюс** шаблонный хелпер `toJsonArray<T>` для единообразной сериализации коллекций. Так закрыты обе строки таблицы ООП — «фабрика/полиморфизм» и «шаблоны».
- **Интерфейсы репозиториев** — кроме `IAssetRepository` явно введены `IDepartmentRepository` и `IMovementRepository` (использовались в сервисах, но не были описаны).
- **`OfficeEquipment`** — метод суммы лет учитывает незавершённый год пропорционально месяцам; по истечении срока службы возвращается полный износ (страховка от ошибок округления).
